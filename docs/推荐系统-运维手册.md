# 推荐系统 · 运维手册

面向"怎么用/怎么调/怎么排查",不讲算法原理(原理见 `推荐系统-设计-Phase1.5-2.md`)。

## 一、三张核心表(均已存在,勿重建)

| 表 | 作用 | 对应 spec |
|---|---|---|
| `user_events` | 用户行为流水(谁 对 哪条内容 做了什么 + duration/时间) | UserAction |
| `product_tags` / `posts.hashtags` | 内容标签(商品用 product_tags;帖子/视频用 hashtags 数组) | ContentTag |
| `user_preferences` | 用户对每个标签的兴趣分(定期重算) | UserPreference |

> `user_events` 通用字段:`target_type`('post'/'product') + `target_id`;商品另存 `product_id` 外键。

## 二、行为权重表(打分用)

集中在 [backend/src/services/recommend/config.js](../backend/src/services/recommend/config.js)。埋点权重在 `recommendationService.js` 的 `EVENT_WEIGHTS`:

| action | 权重 | 说明 |
|---|---|---|
| view | 1 | 点开 |
| click | 2 | 点击 |
| favorite | 4 / like 7 | 收藏 / 帖子赞 |
| comment | 6 | 评论 |
| video_complete | 6 | 完播 |
| video_5s | 3 | 前 5 秒钩子 |
| dwell | 3×归一化 | 停留(除以应有阅读时长) |
| impression | 0 | 仅作分母 |
| skip | −5 | 快速划走(负) |

**时间衰减**:偏好聚合按 `score = Σ(weight × exp(-天数/30))`,只看近 90 天 → 一个月前的行为衰减到 ~37%。

## 三、推荐流程(文字版)

```
埋点(POST /track, 30分钟去重)
        ↓ 写 user_events
偏好聚合(rebuildPreferences, 带衰减)
        ↓ 写 user_preferences
推荐(GET /api/recommend/feed):
   召回(近7天+关注+兜底)
   → 打分 = 质量 × 匹配度 × 新鲜度 − 负反馈
   → 打散(同作者降权)
   → 20% 名额强留给 <24h 新内容
   → 冷启动(偏好标签<3)时兴趣项退化, reason='cold_start'
```

## 四、接口一览

| 接口 | 说明 |
|---|---|
| `POST /api/recommendation/track` | 埋点。body `{ targetType, targetId, eventType, duration? }`,商品旧调用 `{ productId, eventType }` 兼容。30 分钟去重,返回 `{ success, deduped }` |
| `GET /api/recommend` | v1 简易帖子推荐 |
| `GET /api/recommend/feed?limit=20&mediaType=video` | v2 软信号打分,返回 `{ items, reason }` |
| `GET /api/recommend/videos` | 等价 `/feed?mediaType=video` |

## 五、怎么手动刷新偏好

```bash
cd backend
node scripts/rebuildPreferences.js         # 全量重算 user_preferences（带衰减）
```

## 六、怎么调试

```bash
cd backend
# 看软行为采集情况（各行为条数 + dwell 平均秒数）
node scripts/checkSoftBehavior.js

# 看某用户的推荐结果 + 分数拆解 [质 兴 鲜 罚] + 走了 personalized 还是 cold_start
node scripts/previewFeedV2.js 13800000002
```

看某用户的 top 标签(SQL):
```sql
select tag, round(score::numeric, 3) as score
from user_preferences
where user_id = '<userId>'
order by score desc limit 10;
```

推荐为什么是这个结果:`previewFeedV2.js` 每条打印 `[质=quality 兴=interest 鲜=freshness 罚=penalty]`,四项相乘减惩罚即最终分。想调某一项 → 改 `config.js` 对应权重 → 重跑预览对比。

## 七、调参入口

**所有可调数值集中在 [config.js](../backend/src/services/recommend/config.js)**:信号权重、衰减半衰期、新鲜度、兴趣系数、质量系数、惩罚、新内容配额(20%)、冷启动阈值。校准只改这一个文件,不碰逻辑代码。

## 八、定时刷新(Supabase pg_cron)

方案:**pg_cron**。在 Supabase SQL Editor 跑一次 [backend/scripts/setup_pgcron_refresh.sql](../backend/scripts/setup_pgcron_refresh.sql):
- 启用 `pg_cron` 扩展;
- 把衰减聚合封成 DB 函数 `refresh_user_preferences()`;
- 排程每天刷新(`0 19 * * *` UTC = 中国凌晨 3:00)。

运维:
```sql
select * from cron.job;                              -- 看排程
select * from cron.job_run_details order by start_time desc limit 20;  -- 看执行历史
select refresh_user_preferences();                   -- 手动立即跑
select cron.unschedule('refresh_user_preferences_daily');  -- 取消
```

> ⚠️ 该函数刷新的是**商品维度**的 `user_preferences`(join `product_tags`)。**帖子/视频 feed**(`feedRank`)目前是**每次请求实时聚合**(`profile.js`),不依赖这张表,故无需 cron;等量级上来要持久化帖子偏好时再扩。
